## Bugs encontrados y corregidos

### 1. `SQLSERVER_USER` sin guion bajo en GitHub Secret

**Síntoma**: pod `backend` en `CrashLoopBackOff`. Log:
```
Error de inicio de sesión del usuario 'inventarioapp'. (error 18456)
```

**Causa**: el GitHub Secret `SQLSERVER_USER` tenía `inventarioapp` (sin
`_`). El login en SQL Server se llama `inventario_app`.

**Fix permanente**: usuario corrigió el Secret en GitHub.
```
Settings → Secrets → SQLSERVER_USER = inventario_app
```

### 2. Bug en `ldap_service.py` — código incorrecto para Active Directory

**Síntoma**: login respondía `401` incluso con credenciales correctas.
Log del backend:
```
obtener_rol: user_dn=mgomez@itu.local
[error LDAP: No Such Object]
rol=None → 401
```

**Causa raíz — dos problemas en `obtener_rol()`**:

a) El admin bind usaba:
```python
Connection(server, user=f"cn=admin,{settings.ldap_base_dn}", password="admin")
```
Esta cuenta (`cn=admin,DC=itu,DC=local`) no existe en Active Directory.

b) La búsqueda de grupos usaba:
```python
search_filter=f"(member={user_dn})"   # user_dn = "mgomez@itu.local"
```
El atributo `member` de AD guarda DNs completos (`CN=Maria Gomez,OU=User,...`),
no UPNs, así que el filtro nunca matcheaba nada → `rol=None` → `401`.

**Fix correcto** (ver detalle en `active-directory/README.md` sección 4):

`app/core/config.py` — agregar en la clase `Settings`:
```python
ldap_bind_dn: str
ldap_bind_password: str
```

`app/services/ldap_service.py` — reemplazar el bloque de admin bind y
búsqueda de grupos:
```python
# Bind con la cuenta de servicio del configmap/secret
admin_conn = Connection(server, user=settings.ldap_bind_dn,
                        password=settings.ldap_bind_password, auto_bind=True)

# Buscar al usuario por UPN y leer el atributo memberOf (backlink)
admin_conn.search(
    search_base=settings.ldap_base_dn,
    search_filter=f"(userPrincipalName={user_dn})",
    search_scope=SUBTREE,
    attributes=["memberOf"],
)
rol = None
if admin_conn.entries:
    grupos_usuario = admin_conn.entries[0].entry_attributes_as_dict.get("memberOf", [])
    for grupo in ("Tecnicos", "Docentes", "Alumnos"):
        prefijo = f"cn={grupo},".lower()
        if any(dn.lower().startswith(prefijo) for dn in grupos_usuario):
            rol = grupo
            break
```

El ConfigMap ya tiene `LDAP_BIND_DN: svc-inventario@itu.local`; el
Secret necesita `LDAP_BIND_PASSWORD: Inventario!2025`.

⚠️ **Este fix solo está en la imagen local de Minikube** (se parcheó
el fuente en el workspace del runner y se reconstruyó la imagen). La
próxima corrida del workflow hará checkout del código original del repo
del backend y volverá a tener el bug.

**Pendiente crítico**: commitear `app/services/ldap_service.py` y
`app/core/config.py` al repo `Agus-tina/Proyecto-Inventario-EGI`,
rama `backend`, luego volver a disparar el workflow.

### 4. MongoDB arrancaba con datos viejos de un deploy anterior

**Síntoma**: backend lograba conectarse a Mongo pero fallaba con
`UserNotFound` en cada operación. El pod de Mongo arrancaba limpio en
apariencia (PVC recreada), pero las credenciales no matcheaban.

**Causa**: el hostPath provisioner de Minikube guarda los datos en
`/tmp/hostpath-provisioner/inventario/mongo-data` **dentro del
container Docker de Minikube**. Ese directorio no se borra cuando se
elimina el PVC de Kubernetes — solo la vinculación al volumen. MongoDB
solo inicializa `MONGO_INITDB_ROOT_USERNAME/PASSWORD` cuando arranca
con `/data/db` totalmente vacío; si encuentra datos viejos (del deploy
anterior con otro usuario/contraseña), los usa y descarta las variables
de entorno.

**Fix**:
```bash
# 1. Bajar MongoDB para que suelte /data/db
kubectl scale deployment mongo -n inventario --replicas=0

# 2. Borrar los datos viejos dentro del container de Minikube
minikube ssh 'sudo rm -rf /tmp/hostpath-provisioner/inventario/mongo-data'

# 3. Levantar MongoDB — ahora /data/db está vacío, MONGO_INITDB corre
kubectl scale deployment mongo -n inventario --replicas=1

# 4. Verificar que levantó con las credenciales correctas
kubectl logs -n inventario deploy/mongo | grep -i "user\|created\|error"
```

Tener esto en cuenta **cada vez que se resetee Mongo** (borrar PVC,
delete del namespace, etc.): hay que limpiar el hostPath también.
