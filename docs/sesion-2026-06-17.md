# Sesión 2026-06-17 — Reducción de RAM y despliegue

## Problema inicial

Al levantar todo el stack (VirtualBox + Minikube + pods) la PC consumía
toda la RAM disponible. Se identificaron 4 capas sin límites configurados:
VMs de VirtualBox, SQL Server, Minikube y MongoDB.

---

## Cambios aplicados

### 1. VMs de VirtualBox (`infra/scripts/setup-virtualbox-nueva-pc.ps1`)

Se agregó una sección al final del script que limita la RAM de cada VM
después de importarlas (o si ya existen):

| VM | RAM asignada |
|---|---|
| pfSense-Gateway | 512 MB |
| DC01-ITU | 1024 MB |
| SQLServer2022 | 2048 MB |

Para aplicar en el entorno actual (VMs apagadas, desde PowerShell en Windows):

```powershell
$vbm = "C:\Program Files\Oracle\VirtualBox\VBoxManage.exe"
& $vbm modifyvm "pfSense-Gateway" --memory 512
& $vbm modifyvm "DC01-ITU"        --memory 1024
& $vbm modifyvm "SQLServer2022"   --memory 2048
```

---

### 2. SQL Server — límite de memoria (`sql-server-iis/scripts/configurar-memoria-sqlserver.sql`)

Nuevo script creado. SQL Server por defecto consume toda la RAM de la VM.
Se limita a 1536 MB (dejando ~512 MB para Windows Server).

Ejecutar una vez como sysadmin en la VM de SQL Server:

```sql
EXEC sys.sp_configure N'max server memory (MB)', 1536;
RECONFIGURE;
```

Ver el script completo en `sql-server-iis/scripts/configurar-memoria-sqlserver.sql`.

---

### 3. Minikube (`infra/scripts/setup-linuxegi-nueva-pc.sh`)

Se agregaron `--memory=2048 --cpus=2` al comando `minikube start`.

Para aplicar en el entorno actual (desde LinuxEGI), requiere recrear el cluster:

```bash
minikube stop
minikube delete
minikube start --cni=calico --driver=docker --ports=30080:30080/tcp --memory=2048 --cpus=2
```

⚠️ `minikube delete` borra todo el estado, incluyendo el Secret. Ver sección
"Secretos de Kubernetes" más abajo para recrearlo.

---

### 4. MongoDB (`kubernetes/deployments/mongo-deployment.yaml`)

- Agregado `--wiredTigerCacheSizeGB 0.25` (sin esto MongoDB puede consumir
  varios cientos de MB solo de cache).
- Requests: 256Mi → 128Mi
- Limits: 512Mi → 384Mi

---

### 5. Backend (`kubernetes/deployments/backend-deployment.yaml`)

- Requests: 256Mi → 128Mi
- Limits: 512Mi → 256Mi

---

### 6. Runbook actualizado (`docs/runbook-despliegue.md`)

- Comando `minikube start` actualizado con `--memory=2048 --cpus=2`.
- Agregado paso 5 en Fase 2 para ejecutar `configurar-memoria-sqlserver.sql`.

## Problemas resueltos durante la sesión

### DNS en LinuxEGI

La VM no resolvía nombres externos. Fix temporal (se pierde al reiniciar):

```bash
echo "nameserver 192.168.56.2" | sudo tee /etc/resolv.conf
```

Para que sea permanente, configurar el DNS en netplan o en `/etc/systemd/resolved.conf`.

### Backend en CrashLoopBackOff

**Causa:** el Secret de Kubernetes quedó con credenciales vacías porque el
workflow corrió antes de cargar los GitHub Secrets.

**Fix:** recrear el Secret manualmente (ver sección "Secretos de Kubernetes").

### Login SQL Server fallando (error 18456)

**Causa:** el login `inventario_app` no existía o tenía contraseña incorrecta.

**Fix aplicado en la VM de SQL Server:**

```sql
-- Crear login si no existe
IF NOT EXISTS (SELECT 1 FROM sys.server_principals WHERE name = 'inventario_app')
    CREATE LOGIN inventario_app WITH PASSWORD = 'InventarioApp!2025', CHECK_POLICY = ON;

-- Resetear contraseña (si ya existía con otra password)
ALTER LOGIN inventario_app WITH PASSWORD = 'InventarioApp!2025';

-- Crear usuario en la base y asignar roles
USE inventario_ubicaciones;
IF NOT EXISTS (SELECT 1 FROM sys.database_principals WHERE name = 'inventario_app')
    CREATE USER inventario_app FOR LOGIN inventario_app;
ALTER ROLE db_datareader ADD MEMBER inventario_app;
ALTER ROLE db_datawriter ADD MEMBER inventario_app;
ALTER ROLE db_ddladmin   ADD MEMBER inventario_app;
```

## Secretos de Kubernetes

Para recrear el Secret `backend-secret` en LinuxEGI:

```bash
kubectl create secret generic backend-secret -n inventario \
  --from-literal=JWT_SECRET_KEY="$(python3 -c 'import secrets;print(secrets.token_hex(32))')" \
  --from-literal=SQLSERVER_USER='inventario_app' \
  --from-literal=SQLSERVER_PASSWORD='InventarioApp!2025' \
  --from-literal=MONGO_USER='inventario_app' \
  --from-literal=MONGO_PASSWORD='inventario_app' \
  --from-literal=LDAP_BIND_PASSWORD='Inventario!2025' \
  --dry-run=client -o yaml | kubectl apply -f -
```

## GitHub Secrets (pendiente cargar)

Ir a **Settings → Secrets and variables → Actions** en el repo y agregar:

| Secret | Valor |
|---|---|
| `REPO_ACCESS_TOKEN` | Personal Access Token con permiso `repo` sobre `Agus-tina/Proyecto-Inventario-EGI` |
| `JWT_SECRET` | string aleatorio largo (ej. `python3 -c 'import secrets;print(secrets.token_hex(32))'`) |
| `SQLSERVER_USER` | `inventario_app` |
| `SQLSERVER_PASSWORD` | `InventarioApp!2025` |
| `MONGO_ROOT_USER` | `inventario_app` |
| `MONGO_ROOT_PASSWORD` | `inventario_app` |
| `LDAP_BIND_PASSWORD` | `Inventario!2025` |

Sin estos secrets, el próximo deploy del runner arrancará con credenciales vacías.

## Pendientes

### Frontend — API_URL hardcodeada (bloqueante para login)

El archivo `js/api.js` del repo `Agus-tina/Proyecto-Inventario-EGI`
(rama `frontend`) tiene:

```js
const API_URL = "http://localhost:8000"  // ← esto
```

El navegador intenta conectarse a `localhost:8000` en la PC del usuario,
que no existe. El proxy nginx (`/api/` → `backend-service:8000`) queda
sin uso.

**Fix:** cambiar en el repo del frontend:

```js
const API_URL = "/api"
```

Luego hacer commit, push y re-disparar el workflow para rebuild de la imagen.
